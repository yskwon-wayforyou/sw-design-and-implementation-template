# 폴더 단위 이해용 문서 작성

에이전트가 사용자를 **이해시키기 위해** 만드는 문서·다이어그램·설계 노트는, 한 파일에 몰아넣기보다 **폴더 단위로 경계**를 나누고, 그 안에서 파일 역할을 고정한다.

## 언제 적용하는가

- 기능·아키텍처·모듈 경계를 **처음부터 설명**할 때
- 내용이 길어질 것이 예상될 때(유스케이스 묶음, 품질 시나리오, 단계별 워크플로)
- 사용자가 **폴더로 정리**, **단계별로 보고 싶다**고 할 때

## 폴더 설계 원칙

1. **한 폴더 = 한 응집 주제** (예: 한 기능, 한 단계, 한 품질 영역). 주제가 섞이면 폴더를 나눈다.
2. **깊이 2~3단**을 기본으로 한다. `주제/세부.md` 또는 `주제/하위주제/파일.md`. 더 깊어지면 상위에 요약 인덱스를 둔다.
3. **경로 이름**은 OS·도구 호환을 위해 `kebab-case` 또는 짧은 영문 식별자를 권장한다. 한글 제목은 파일 **첫머리 `# 제목`** 으로 쓴다.
4. **기존 구조**가 있으면(예: `{프로젝트 루트}/docs/adr`, `docs/quality/` 등) 그 규칙을 깨지 않고 **확장**한다. 새 묶음이 필요하면 상위에 한 단계 폴더만 추가한다. 설계·이해용 산출은 원칙적으로 **`{단위 프로젝트 루트}/docs/`** 아래에 둔다.

## 표준 파일 역할 (권장)

| 파일 | 역할 |
|------|------|
| `README.md` 또는 `00-overview.md` | 이 폴더의 목적, 읽는 순서, 다른 파일 링크 |
| `architecture.md` / `structure.md` | 경계, 컴포넌트, 트레이드오프 (필요 시만) |
| `flows.md` 또는 주제별 `*.md` | 시퀀스·상태는 아래 다이어그램 규칙 준수 |
| `glossary-local.md` | 이 묶음에서만 쓰는 용어 (선택) |

한 폴더에 **엔트리 파일은 하나**만 두는 것을 권장한다 (`README.md`가 엔트리).

## 다이어그램·코드

- 구조·흐름은 사용자 규칙에 따라 **Mermaid**로 작성한다.
- 다이어그램이 많으면 `diagrams.md` 한 파일에 모으거나, 흐름별로 `flow-주문.md` 처럼 **최소 분할**한다.
- 예시 코드는 **짧은 스니펫**만; 긴 구현은 별도 패키지·모듈 경로를 가리키고 설명 문서에는 링크만 둔다.

## 작성 순서 (워크플로)

1. 상위 주제 폴더 경로 결정 (기존 트리와 충돌 없는지 확인).
2. **엔트리 `README.md`** 먼저 작성: 목적, 대상 독자, 권장 읽기 순서, 하위 파일 목록.
3. 세부 파일을 **읽기 순서대로** 채운다 (개요 → 경계 → 흐름 → 결정 사항).
4. 마지막에 엔트리의 링크·순서를 다시 맞춘다.

## 엔트리 템플릿

엔트리 `README.md`에 아래 블록을 채워 넣는다.

```markdown
# [주제 한 줄]

## 이 폴더에서 답하는 질문
- …

## 읽는 순서
1. …
2. …

## 문서 목록
- [파일명](./파일명.md) — 한 줄 설명

## 관련 코드·설정 (있으면)
- `path/to/...`
```

## 안티 패턴

- 루트에 설명 파일만 수십 개 나열하고 **인덱스 없음**
- 동일 내용을 여러 파일에 **복붙** (한 곳에 두고 링크)
- 설명 없이 **폴더만** 만들고 비어 둠

## 다른 규칙과의 정렬

프로젝트에 `collaborative-design-first` 규칙이 있으면: **다이어그램·구조 설명을 이 스킬 형태로 폴더에 담은 뒤**, 사용자가 이해·허가했을 때만 해당 모듈 구현으로 넘어간다.

## 검증 체크리스트

- [ ] 엔트리 파일이 있고, 읽는 순서가 명시됨
- [ ] 각 하위 파일이 **한 주제**에 고정됨
- [ ] 상대 경로 링크가 동작함
- [ ] Mermaid/코드 펜스가 렌더 규칙에 맞음 (펜스는 단독 줄에서 시작)
