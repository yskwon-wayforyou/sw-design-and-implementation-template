# ADR 작성

구조적 의사결정(**Architectural Decision**)은 시간이 지나면 맥락이 사라진다. ADR은 **당시의 맥락·선택·결과**를 짧은 텍스트로 남겨, 온보딩·감사·되돌리기 비용을 줄인다.

## 언제 ADR을 쓰는가

다음 중 하나라도 해당하면 **새 ADR**을 고려한다.

- 여러 **후보 구조**가 있었고 하나를 택했다 (트레이드오프가 있음).
- **경계·저장소·통신·보안·관측** 등 변경 시 비용이 크다.
- 팀/조직이 **표준에서 벗어난** 선택을 했다 (예외가 정당해야 함).
- 기존 ADR을 **대체**한다 (상태·링크로 처리).

반대로, 단순 리팩터·버그 수정·명백한 표준 적용만으로는 ADR이 **필수는 아님**. 필요하면 한 줄짜리 “Non-decision” 노트로 충분할 수 있다.

## 보관 위치와 파일명

- 기본: 저장소 루트 또는 문서 루트 아래 `docs/adr/` (프로젝트에 이미 `adr/`가 있으면 그쪽을 따른다).
- 파일명: `NNNN-short-title-kebab-case.md` (NNNN은 4자리, **0 패딩**). 예: `0001-use-json-file-store.md`.
- 한 ADR = 한 파일. 길어지면 **부록 파일**을 같은 번호 접두로 분리한다 (`0001-appendix-options.md`).

## 상태(Status)

| 값 | 의미 |
|----|------|
| Proposed | 검토 중 |
| Accepted | 채택됨 |
| Deprecated | 더 이상 권장하지 않음 |
| Superseded by ADR-XXXX | 후속 ADR로 대체됨 |

상태를 바꿀 때는 **본문 상단 메타데이터**와 마지막에 **변경 이력 한 줄**을 남긴다.

## 본문 템플릿 (필수 섹션)

에이전트는 아래 순서를 지킨다. 프로젝트에 이미 ADR 템플릿이 있으면 **그 템플릿을 우선**한다.

```markdown
# NNNN. [짧은 제목]

## 메타데이터
- 상태: Proposed | Accepted | Deprecated | Superseded by [ADR-XXXX](./XXXX-....md)
- 날짜: YYYY-MM-DD
- 결정자: (역할 또는 이름, 알 수 있으면)
- 관련: (이슈, PR, use case ID, QS ID 등 링크)

## 맥락 (Context)
무엇이 문제였는가. 제약(일정, 규제, 운영, 팀 역량)을 구체적으로.

## 결정 요인 (Decision drivers)
우선순위가 있던 품질 속성·비즈니스 목표 (예: 정합성 > 지연).

## 고려한 대안 (Options considered)
**이름 붙은 대안 ≥ 3개** (아래 “대안 최소 3개” 규칙). 각 대안을 한 단락 이상으로, 왜 **기각** 또는 **보류**였는지 적는다.

## 결정 (Decision)
선택한 방안을 **단정적으로** 한두 문단으로.

## 결과 (Consequences)
긍정·부정·중립을 나누어 적는다. 알려진 기술 부채·후속 ADR 후보를 명시.

## 준수·검증 (Compliance / validation) — 선택
어떻게 “이 결정이 지켜지는지” (리뷰 체크리스트, 테스트, 메트릭).

## 다이어그램 — 선택
구조를 한눈에 보여줄 필요가 있으면 Mermaid로 **한 개**를 권장. 복잡하면 부록으로 분리.
```

## 대안 최소 3개 (기본 규칙)

**목적**: 최적 설계에 가깝게 가려면 비교 가능한 **후보 공간**을 의도적으로 넓힌다.

1. **“고려한 대안”에는 서로 다른 방향의 대안을 최소 3개** 둔다. 각 대안에는 짧은 **이름**(예: `A-단일-DB-낙관적-락`)을 붙인다.
2. **실질 후보가 2개뿐**이면, 비교 가치가 있는 제3안을 **의도적으로** 추가한다. 허용되는 예시:
   - **현상 유지** / **결정 연기**(범위·일정 축소)
   - **단계적 도입**(하이브리드·이중 쓰기·기능 플래그)
   - **외부 위임**(매니지드·SaaS·게이트웨이 일원화)
   - **다른 일관성·경계 모델**(같은 기술 스택 안에서도 아키텍처 의미가 달라지는 변형)
3. **정말로 3개를 채울 수 없다고 판단되면**(극히 드묾): `고려한 대안` 맨 앞에 **“예외: 대안 <3개인 이유”** 소단락을 두고, 탐색을 막은 **하드 제약**(규제, 단일 벤더, 고정 계약)을 나열한다. 그 경우에도 **비교용 제3안**(예: 연기·축소 범위)을 가능한 한 기술한다.
4. 필요하면 부록 파일 `NNNN-appendix-options.md`에 비교표(드라이버별 점수, 거친 추정)를 둔다.

## 작성 절차

1. **번호**: 폴더 내 최대 번호 +1 을 사용한다. 충돌 시 사용자에게 확인.
2. **초안**: 상태 `Proposed`로 전 섹션 채운다. **대안 ≥3 검토**를 먼저 채운 뒤 결정을 쓴다. 빈 “결정”은 두지 않는다.
3. **리뷰 반영**: 맥락·대안·결과를 보강; 모호한 용어는 프로젝트 용어집과 맞춘다.
4. **채택**: 합의 후 `Accepted`로 바꾸고 날짜를 확정한다.
5. **대체**: 새 ADR에서 이전 ADR을 `Superseded by`로 연결하고, 구 ADR 상단도 갱신한다.

## 품질 기준

- **한 ADR = 하나의 결정** (여러 무관한 결정을 한 파일에 넣지 않는다).
- 맥락은 “그때 왜 그랬는지”가 **낯선 사람**에게도 읽히게 쓴다.
- **대안 ≥3** 규칙을 만족하는지 스스로 검사한다. “당연히 이거다”만 있으면 **연기·축소·현상 유지**라도 제3안으로 올려 트레이드오프를 드러낸다.
- 측정 가능한 전제가 있으면 **수치 또는 출처**를 적는다 (없으면 “미정”이라고 적는다).

## 안티 패턴

- 구현 세부(함수명·라인)만 길게 적고 **결정과 트레이드오프가 안 보임**
- **대안이 2개 이하**인데 예외 소단락 없이 채택으로 넘어감 (탐색 부족으로 읽힘)
- Accepted 이후 본문을 **과거처럼 수정**해 역사를 지움 → 대신 새 ADR로 Amend/Supersede
- `TBD`만 있고 **결정 문장이 없음**

## 프로젝트와의 연결

- `usecases.md`의 핵심 ASR, `quality/scenarios.md`의 QS와 연결할 수 있으면 메타데이터 `관련`에 적는다.
- `collaborative-design-first` 규칙이 있으면: 사용자가 설계를 이해·허가한 뒤 채택된 내용을 ADR로 **고정**한다.

## 완료 전 체크리스트

- [ ] 번호·파일명 규칙 준수
- [ ] 상태·날짜·관련 링크 존재
- [ ] Context / Options / Decision / Consequences가 채워짐
- [ ] **고려한 대안 ≥3** (또는 예외 소단락 + 가능한 한 3안 기술)
- [ ] 대체 관계가 있으면 양방향 링크 확인
