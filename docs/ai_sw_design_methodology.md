# AI 전문가(SWO) 관점 소프트웨어 설계·구현·테스트 방법론

본 문서는 **일반 구조 설계 템플릿**(`design-principles` 워크플로, NFR/QA, Utility Tree)을 전제로, **AI·ML·LLM·RAG·에이전트**가 포함된 소프트웨어에 필요한 개념을 정리한다. 내용은 내부 강의 자료「AI Specialist 설계」PDF 체계(추론/운영 이중 QA, 품질 Trade-off, 가드레일, 침묵하는 실패, 응용 유형별 우선순위 등)에 맞추어 **요약·재구성**하였다.

**정본 역할**: `.cursor/rules/design-principles-ai-system-quality.mdc`, `.cursor/skills/ai-specialist-system-design/SKILL.md`, 품질 관련 에이전트 RULE이 본 문서와 용어를 맞춘다.

---

## 1. 왜 “AI SW”가 일반 SW와 다른가

- **확률적 엔진**: 동일 입력에 분산된 출력; “모름” 대신 **그럴듯한 오답**이 나올 수 있다.
- **지능의 노화(Model decay)**: 학습 이후 데이터 분포 변화에도 관성 출력 → **드리프트** 관리가 필수.
- **과신(Overconfidence)**: 낮은 정확도에도 높은 확신 표시 가능 → **교정·임계·거부 옵션** 설계.
- **침묵하는 실패(Silent / Salient failure)**: 크래시 없이 잘못된 결과만 지속 → **가드레일·모니터링·사후 감사**가 아키텍처 요소다.

따라서 품질은 “응답 시간·가용성”만으로 부족하고, 아래 **이중 초점**으로 나누어 시나리오화한다.

---

## 2. 품질 관점 두 축: 추론 중심 vs 운영 중심

### 2.1 추론 중심(Focused) QA — 학습·추론 경로

다양한 상황에서 **개인정보 보호·공정성·정확한 추론**을 **효율적으로** 달성하는 능력.

| 속성 | 정의 요지 | 설계에 남길 것(예시) |
|------|-----------|------------------------|
| 기능 정확성 | 의도한 정답·정밀도 | 태스크별 지표(정밀도/재현율/F1, BLEU, 등); 인간 평가 샘플링 |
| 강건성 | 악조건에서 정확성 유지 | OOD·적대적 입력·센서 노이즈 시나리오; 감소 학습 |
| 프라이버시 | 부당·불법적 개인 데이터 처리 방지 | 최소 수집, 익명화, PETs(차분 프라이버시 등), 연합학습 검토 |
| 공정성 | 보호 속성 기반 차별 방지 | 측정 그룹, 지표(인구통계 패리티 등), 완화 파이프라인 |
| 수행 효율성 | 자원 대비 시간·처리량 | 지연 p95/p99, GPU 메모리, 배치 처리량, 엣지 제약 |

**설계 관점(추론 파이프라인)**: 학습 데이터셋 구축 → 모델 학습; 데이터 준비 → **입력 가드레일** → 모델 서빙 → **출력 가드레일** → 모델 운영(모니터링·승격·롤백).

### 2.2 운영 중심(Focused) QA — 현장에서 신뢰를 유지·회복

모델을 **실제 현장**에 둘 때, 추론을 **강화(Enhance)** 하는 QA.

| 속성 | 정의 요지 | 설계에 남길 것(예시) |
|------|-----------|------------------------|
| 기능 적응성 | 환경 변화에 적응 | 드리프트 탐지, 재학습·프롬프트·RAG 갱신 주기, 온디바이스 한계 |
| 제어 가능성 | 인간·외부 에이전트 개입 | 킬 스위치, 승인 단계, 롤백, 정책 엔진 override |
| 설명 용이성 | 이유·타당성 표현 | 출처 표시(RAG), 요약 근거, 고영향 시 법적 설명 범위 |

## 3. 침묵하는 실패와 방어 계층

| 구분 | 명시적 실패 | 침묵하는 실패 |
|------|-------------|----------------|
| 징후 | 에러 로그, 크래시 | 로그 정상·서비스 유지·“그럴듯한” 오출력 |
| 발견 | 즉시 | 사고 후 또는 정밀 분석 시 |
| 대응 | 재시작·패치 | 가드레일, 모니터링, 재학습, 도메인 검증 |

**아키텍처적 방어(Safety Net)**:

1. **입력 가드레일**: OOD·저품질·악의적 입력 차단(Prevention).
2. **출력 가드레일**: 상식·정책·윤리·프라이버시 최종 검증(Validation).
3. **모델 운영**: 드리프트·확신도·데이터 품질 지속 관측(Detection & Action).

이 세 층은 `docs/architecture.md` 또는 `docs/domain/model.md`의 컴포넌트로 **이름 붙은 박스**가 되어야 한다.

## 4. 데이터 품질

학습·추론 입력의 품질은 **공정성·강건성·정확성의 상한**을 결정한다.

- 대표성, 라벨 노이즈, 수집 편향, 시계열 갭, PII 혼입.
- Utility Tree에 **“데이터 품질”** 분기를 두고, QS로 측정(재현 가능한 전처리·검증 파이프라인).

## 5. 응용 유형별 QA 우선순위 (요약 표)

과제 도메인에 맞게 `docs/ai_quality_profile.md` 또는 `qualities.md` 서두에 **한 표**로 고정한다. (상/중/하는 강의 예시 우선순위이며, 프로젝트에서 재합의한다.)

| AI 응용 유형 | 강건성 | 프라이버시 | 공정성 | 기능 적응성 | 제어 가능성 | (추론 측 정확성·효율은 별도 QS로) |
|--------------|--------|------------|--------|-------------|-------------|
| 초개인화·추천 | 하 | 상 | 상 | 상 | 하 |
| 인지·지각(자율주행 등) | 상 | 중 | 하 | 중 | 상 |
| 생성형·거대 모델 | 중 | 상 | 상 | 중 | 중 |
| 예측·예지 보전 | 상 | 하 | 하 | 중 | 중 |
| 의사결정·에이전트 | 상 | 중 | 중 | 상 | 상 |

## 6. 품질 Trade-off (아키텍트가 문서화해야 할 충돌)

대표 충돌(완화 전략은 ADR·후보 구조에):

- **품질 강화 vs 수행 효율성**: 앙상블·필터·PETs·XAI·가드레일이 지연·비용 증가.
- **공정성 vs 기능 정확성**: 재샘플링·보호변수 배제·임계 조정이 정밀도·오탐에 영향.
- **프라이버시 vs 기능 정확성**: 익명화·DP·최소화가 유틸리티 하락.
- **설명 용이성 vs 기능 정확성**: 블랙박스 극대화 vs 해석 가능 모델; 사후 설명의 충실도 한계.

**의사결정 절차(요약)**:

1. 도메인별 **필수 QA vs 부가 QA** 구분(고영향·안전·준법 잠금).
2. 선택한 아키텍처가 다른 QA에 미치는 **부정적 영향** 기록.
3. XAI·PETs·압축·가드레일 등 **완화**를 후보로 비교.

원칙: **모든 QA를 동시에 최대로 만족하는 단일 설계는 없다.** “허용 가능한 조합”을 근거와 함께 고른다.

## 8. 테스트·검증 시 유의

- **확률 출력**: 점 추정만이 아니라 분포·확신·거부 구간 검증.
- **회귀**: 데이터 버전·모델 버전·프롬프트·RAG 코퍼스 버전을 **재현 가능**하게 고정.
- **공격·오용**: Jailbreak, 데이터 추출, 프롬프트 인젝션을 보안 시나리오에 포함.
- **상용**: `commercial-product-design-requirements` 및 `prepare-testing` / `prepare-deployment` 커맨드와 정합.
